# Hotfix Branch Impact

## Inputs

优先接收：

```text
Base: <commit / tag / GitHub commit 链接>
Branch: <branch-name>
Commit: <head commit / GitHub commit 链接>
```

可选：

```text
Branches:
- <branch-a>
- <branch-b>
```

规则：
- 优先使用完整 SHA；短 SHA 仅在可唯一解析时使用。
- GitHub commit 链接先解析为完整 SHA。
- `Base` 表示上一个最终包或基线版本；`Commit` 表示本次提测头或最终 release 头。
- 若 `Base` 或 `Commit` 无法解析，不要硬猜，直接要求补充完整 SHA 或 commit 链接。

## Workflow

1. 校验并解析 `Base`、`Commit`。
2. 分析 `Base...Commit` compare，重点看 commits、files、路径分布、关键 commit message。
3. 按模块与链路归纳实际影响，不逐文件复述 diff。
4. 默认每条合并输出“改动概括 + 覆盖点”；仅当用户明确只要影响范围时，省略覆盖点。
5. 公共层、基础层、构建/发布链路改动单独归纳；热更新或最终 release 默认补“热更新/发版专项”。
6. 若用户要求代码块格式，则正文必须且只能放在单个代码块中。
7. 若只给了分支列表、未给最终 head，可先输出分支集合视角的预估范围，并明确最终以 release head 为准。

## Output

默认直接输出：

```text
影响范围 + 测试覆盖点

1. 模块A：<一句话概括改动>
- 覆盖：<测试点1> / <测试点2> / <测试点3>

2. 模块B：<一句话概括改动>
- 覆盖：<测试点1> / <测试点2> / <测试点3>

3. 通用/基础层：<一句话概括改动>
- 覆盖：<测试点1> / <测试点2> / <测试点3>

4. 热更新/发版专项
- 覆盖：<专项点1> / <专项点2> / <专项点3>
```

如果用户明确只要影响范围，则输出：

```text
影响范围

1. 模块A
- <影响点1>
- <影响点2>

2. 模块B
- <影响点1>
- <影响点2>
```

要求：
- 保持简洁，优先 4–8 条。
- 覆盖点优先写主流程、状态切换、边界、异常、切换场景。
- 若只有工程或发布改动，没有直接业务测试点，就写成构建/流程验证类覆盖。
- 若要求代码块格式，不要在代码块外补任何前言、解释或总结。

## Checklist

发送前检查：
1. `Base` 和 `Commit` 可真实解析。
2. 结论基于 compare 结果，而不是凭空猜测。
3. 输出按模块归纳，而不是按文件堆砌。
4. 默认已合并“影响范围 + 测试覆盖点”；若用户只要影响范围，已去掉覆盖点。
5. 结果足够简洁，可直接转发到 Slack。
