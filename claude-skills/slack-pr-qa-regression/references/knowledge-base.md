# Slack PR QA Regression Knowledge Base

用于沉淀用户对“根据 PR 生成回归项”场景的已确认反馈、常见误判和生成约束。

## 使用规则

1. 每次生成回归项前，先读取本文件。
2. 若本文件中的历史反馈与当前 PR 场景相关，优先遵守本文件，避免重复犯错。
3. 当用户指出“描述不准确 / 理解偏差 / 覆盖遗漏 / 格式错误”时，在确认后追加到本文件。
4. 对于用户明确指出的问题，默认视为需要自动沉淀；除非用户明确说“不用记录”。
5. 优先记录：
   - 错误模式
   - 用户纠正
   - 后续生成约束
   - 推荐措辞
   - 适用范围
   - 错误分类

## 错误分类模板

后续新增反馈时，优先按以下分类归档：

- `格式错误`：标题不对、没放代码块、代码块外有正文、混入前言/总结/步骤等
- `业务理解错误`：对 PR 意图、业务规则、状态流转、平台差异理解偏差
- `测试设计不可执行`：写成测试中无法实际构造/无法人工验证/不符合真实使用方式的用例
- `覆盖遗漏`：漏掉主流程、边界、异常、多端、切换、回归中的关键维度
- `措辞不当`：表述不符合团队习惯、过度实现细节化、容易误导执行方

## 已确认反馈

### [2026-03-26] 硬件连接标识类 PR 不要写成“修改大小写”的测试

- 错误分类：业务理解错误 / 测试设计不可执行
- 适用范围：硬件连接、BLE / USB、设备标识识别、跨会话识别类 PR
- 错误模式：把回归项写成“修改 ConnectID / UUID 大小写后校验结果是否正确”。
- 用户纠正：ConnectID 和 UUID 都是硬件返回的，App 端存储的也无法修改，不涉及大小写变更测试。
- 后续生成约束：
  - 不要把此类回归项写成手动改 connectId / UUID / 大小写的测试。
  - 应围绕“同一硬件在不同连接会话、断连重连、重启 App 后，是否仍能正确识别已有钱包/设备”来设计用例。
  - 若 PR 文案提到 connectId change between sessions，应优先抽象成“跨会话设备标识变化后的识别结果”，而不是直接展开成可人工操作的字段修改。
- 推荐措辞：
  - `同一硬件断连重连后再次进入流程，仍可正确识别已存在的标准钱包`
  - `重启 App 后重新连接同一硬件，仍可正确命中已有设备对应的钱包`
  - `多设备场景下，仅按当前连接硬件识别，不误命中其他设备`

### [2026-04-02] Send / 转账类 PR 需要区分“改动命中链路”与“高风险安全主流程”，不要把 memo 覆盖写成全链逐条转账

- 错误分类：覆盖遗漏 / 测试设计不可执行
- 适用范围：Send、转账、收款人选择、地址簿、最近收款人、memo/tag、金额校验、安全相关 PR
- 错误模式：
  - 把 memo 网络覆盖笼统写成“有 memo 信息的链都测一遍”。
  - 对转账主流程只写粗粒度用例，没有把高风险校验点和安全相关阻断点展开。
- 用户纠正：
  - 有 memo 信息的网络很多，不适合默认要求每条链都完整转账一遍。
  - 转账属于重要且偏安全的主流程，回归项需要列得更细，覆盖关键校验、阻断、回填、异常和确认链路。
- 后续生成约束：
  - memo/tag 覆盖默认采用“规则 + 代表链”方式：先覆盖组件/流程级行为，再挑 2~3 条典型 memo 网络验证，不默认扩展到每条链全量转账。
  - 若 PR 涉及 Send 主流程、收款人、地址校验、金额校验、确认页、地址簿、最近收款人，必须把高风险主流程细化，覆盖进入路径、回填、校验、阻断、返回重选、确认提交、成功/失败分支。
  - 对安全相关环节优先补充：错误地址阻断、无效金额阻断、余额不足阻断、必填 memo/tag 阻断、跨页状态一致性、确认页信息一致性、异常返回后数据保持。
- 推荐措辞：
  - `memo/tag 能力按代表链覆盖：例如 XRP / TON / Stellar（或 Cosmos 生态带 memo 场景），验证录入、回填、展示、校验与历史详情一致性`
  - `转账高风险链路需细化覆盖：收款人选择、金额输入、确认页、签名广播、失败返回、重新编辑、结果落库`
  - `优先把“应阻断”的错误场景写清楚，而不是只写成功发送主流程`
- 进一步固定要求：后续遇到类似 Send / 转账 / 安全高风险流程相关 PR，默认按本次增强版粒度输出回归项；即在固定代码块格式下，优先细化安全校验、阻断、回填、确认页一致性、失败返回与代表链 memo/tag 覆盖，不再仅给粗粒度主流程用例。
