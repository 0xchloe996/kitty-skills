# Slack PR QA Regression

## Workflow

1. 读取 PR。若当前消息或线程首条消息已包含 PR 链接，直接使用，不再追问。
2. 生成前读取 [`references/knowledge-base.md`](references/knowledge-base.md)，优先规避已知错误模式与已确认反馈。
3. 按模块整理回归项，默认优先覆盖：主流程、边界、异常、状态切换、多端差异、关联回归。
4. 对存在明确判定逻辑、分档规则、组合条件或状态映射的模块，优先使用“规则 + 具体示例”的简洁写法。
5. 保持简洁；除非用户明确要求，不展开公式推导、计分过程或实现细节。
6. 若用户在当前对话中指出错误，修正后同步更新 `references/knowledge-base.md`，除非用户明确要求不要记录。

## Output

- 标题优先使用 `PR #xxxx QA 回归项`；拿不到 PR 号时退化为 `QA 回归项`。
- 正文必须且只能放在**单个代码块**中。
- 代码块内只允许：分节标题、编号用例列表。
- 不写：前言、总结、前置条件、操作步骤、风险说明、额外解释。
- 在 Slack thread 中被 @ 时，优先回复到同一 thread。
- 不输出控制标记或铺垫语，如“我先看 PR”“以下是回归项”“已按要求生成”。

## Template

```text
PR #xxxx QA 回归项

一、模块A
1. xxx
2. xxx

二、模块B
3. xxx
4. xxx

三、回归
5. xxx
6. xxx
```

## Checklist

发送前检查：
1. 已读取 PR，而不是凭空猜测。
2. 已读取 `references/knowledge-base.md`。
3. 标题正确。
4. 正文完整放在一个代码块中。
5. 代码块外没有多余正文。
6. 只写用例，并尽量按模块分节。
7. 已覆盖主流程、边界、异常、切换、多端或关联回归中的 relevant 部分。
8. 若本轮刚收到用户纠错，已在修正后同步更新知识库（除非用户要求不记录）。

若第 4 或第 5 项不满足，直接重写后再发送。
