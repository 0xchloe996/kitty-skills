# Global Clean Code Rules

适用于所有项目的高质量、精简代码规则。AI 必须先识别当前项目技术栈，再按对应最佳实践写代码。

## 总体目标

- 在任何项目中，你都必须写出简洁、可读、可维护、可测试的高质量代码。
- 在开始执行任务前，先快速识别当前项目的技术栈和结构，并据此选择合适的实现方式。

## 识别当前项目技术栈

在处理任务前，你应当：

1. 通过文件和配置自动判断技术栈，例如：
   - 存在 `package.json` / `tsconfig.json` / `pnpm-lock.yaml` → Node.js / TypeScript 生态。
   - 存在 `next.config.js` / `app/` / `pages/` → Next.js。
   - 存在 `expo` / `app.json` → React Native / Expo。
   - 存在 `pom.xml` → Java / Spring。
   - 存在 `pyproject.toml` / `requirements.txt` → Python 项目。
   - 其他技术栈请根据常见配置文件和目录结构自行推断。
2. 根据识别到的技术栈，遵循该生态主流的社区最佳实践（目录结构、状态管理方式、Testing 框架等），而不是混用不适配的工具或风格。

如果不确定技术栈或框架版本，应在回答中先说明你的推断，再谨慎行动。

## 通用编码规则（适用于所有语言）

### 1. 函数与模块
- 函数/方法/组件应尽量保持单一职责和短小体积，避免"上百行大函数"。
- 避免深层嵌套的条件和循环；复杂逻辑优先拆分成多个小函数，或通过中间变量表达意图。

### 2. 命名与可读性
- 使用语义明确的命名：函数用动词短语，变量名表达含义而非类型，如 `fetchUser`, `totalPrice`, `isLoading`。
- 禁止使用 `tmp`, `data1`, `foo`, `bar`, `doStuff` 这类模糊命名，除非是非常局部且寿命极短的变量。
- 优先通过清晰的代码结构表达意图，其次才是注释；不要用注释掩盖糟糕的结构。

### 3. 避免重复（DRY）
- 遇到重复或高度相似的代码时，优先提取成公共函数、类、Hook、utility 模块或共享组件，而不是复制粘贴。
- 抽象时保持 API 简单清晰，不要为了复用而进行过度抽象。

### 4. 错误处理与边界情况
- 对所有外部输入、IO、网络请求、异步操作处理好错误和边界情况，避免未捕获异常。
- 能用类型系统或校验器防止错误时，优先使用类型/schema，而不是依赖后期修补。

### 5. 测试与安全网
- 为新功能和 bug 修复编写或更新测试，用项目常用的测试框架（根据当前项目自动判断，如 Jest / Vitest / Playwright / pytest 等）。
- 在说明中列出应该运行的测试命令，并在用户要求时给出具体示例。
- 修改任何公共 API 或核心逻辑时，优先扩充测试覆盖率，而不是仅进行手动验证。

### 6. 与现有代码对齐
- 保持与当前项目已有代码的风格、模式和依赖一致，不要引入与现有栈不兼容的新库或新模式，除非用户明确同意。
- 当现有代码明显不合理但改动范围太大时，可以在局部做更合理的实现，并在说明中指出潜在的重构建议。

## 工作流程

### 7. 先给计划，再写代码
- 对于复杂任务（非简单一行改动），先用列表形式给出简短计划：
  - 要查看哪些文件；
  - 打算修改哪些模块；
  - 潜在风险点；
  - 是否需要抽象公共代码或改动公共模块。
- 只有在计划合理之后，再进行大规模修改。

### 8. 小步提交、局部修改
- 只修改与当前需求直接相关的代码，避免无关重构或对大量文件进行格式化。
- 如必须进行重构，保持小步迭代，并在说明中清楚划分"功能改动"和"重构改动"。

### 9. 自动自查清单（每个任务结束前强制执行）

**除非用户明确说「不需要自查」，否则你处理任何非 trivial 任务（修 bug、写功能、重构等）时，必须在结束前自动执行以下自查，并在回答中用「自查结果：」开头逐条输出：**

1. **精简度**：这段代码是否足够简洁？是否有过长函数（> 40 行）或职责过多的组件？如果有，已拆分了吗？如果没拆分，理由是什么？
2. **复用**：是否有重复逻辑或可抽象的公共函数/Hook？如果有，已抽取了吗？如否，理由是什么？
3. **可读性**：是否有深层嵌套、一行超长表达式、模糊命名（tmp/data1/doStuff）？这些已优化了吗？
4. **测试**：关键逻辑是否配了测试？建议用户运行什么命令验证？（给出具体命令示例）
5. **进一步优化**：如果代码还能再精简或更清晰，你会怎么改？给出 1-2 句具体优化建议。

## 默认行为

- 在任何项目中，如果用户没有特别说明例外情况，你必须默认遵守本规则，并根据当前项目的技术栈和结构选择最合适的实现方式。
- 如果项目已经存在本地的 `.cursor` / 项目级规则，则在遵守这些项目特定规则的前提下，再应用本全局规则。
